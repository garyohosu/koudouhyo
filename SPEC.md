# 行動予定表システム 仕様書

## 1. システム名

- システム名: 行動予定表
- リポジトリ名: `koudouhyo`
- 実行ファイル名: `Koudouhyo.exe`
- データベース名: `koudouhyo.db`

## 2. システム概要

本システムは、社員の所在状況を一覧表示し、外線電話応対時に対象者の状況をすぐに確認できるようにするための社内用デスクトップアプリケーションである。

従来の手書きホワイトボードおよび磁石札による運用を電子化し、視認性、更新性、保守性を向上させることを目的とする。

本システムの一覧画面は、現行の行動予定表の見た目および列構成を可能な限り維持し、利用者が違和感なく移行できることを重視する。

## 3. 目的

- 外線電話応対時に、対象者の所在や状況を即座に確認できるようにする
- 出社中か退社済みかを一目で判別できるようにする
- 休暇、出張、外出等の不在理由をすぐ伝えられるようにする
- ホワイトボードへの手書きや磁石札の付け替え作業を削減する
- 更新内容を履歴として保持し、確認できるようにする

## 4. 現行運用の電子化方針

現行運用では以下の情報を表示している。

- 氏名
- 行先
- 内線番号
- 備考
- 状態を示す磁石札
- 氏名札の色による出社／退社表示

電子化後も以下の考え方を維持する。

- 氏名札は白または赤で表示する
- 白は出社中、赤は退社済みを表す
- 休暇、出張、外出等は磁石札に相当する状態表示で表す
- 表の列構成は可能な限り現行の並びを維持する

## 5. システム構成

- 開発言語: Python
- 画面: デスクトップGUI
- 配布形式: exe
- データベース: SQLite
- DB配置: 社内共有サーバー
- 運用: 複数人閲覧、編集は低頻度
- 更新方式: 起動時に最新版確認を実施する

## 6. 共有フォルダ構成

本システムは、共有サーバー上のルートフォルダを1つだけ設定値として持つ。  
データベース、バックアップ、ロックファイル、更新情報、配布用アプリは、この共有ルート配下の固定フォルダに配置する。

例:

`\\server\share\koudouhyo\`

標準構成:

- `app` : 配布用アプリ保存先
- `data` : SQLiteデータベース保存先
- `backup` : DBバックアップ保存先
- `lock` : 編集ロックファイル保存先
- `update` : バージョン情報保存先

想定パス例:

- `\\server\share\koudouhyo\app\current\Koudouhyo.exe`
- `\\server\share\koudouhyo\data\koudouhyo.db`
- `\\server\share\koudouhyo\backup\`
- `\\server\share\koudouhyo\lock\edit.lock`
- `\\server\share\koudouhyo\update\latest.json`

運用ルール:

- 利用者は常に `app\current\Koudouhyo.exe` を起動する
- `app\releases` は管理者向けの版管理・保管用とし、一般利用者は直接起動しない

## 7. 共通設定

設定値は最小限とし、初版は以下の項目のみとする。

- `shared_root`
- `admin_users`

例:

```json
{
  "shared_root": "\\\\server\\share\\koudouhyo",
  "admin_users": ["yamada", "suzuki"]
}
```

アプリは `shared_root` から以下を自動決定する。

- DBパス
- ロックファイルパス
- バックアップ保存先
- 更新情報ファイルパス
- 配布アプリ格納先

## 8. 運用前提

- 閲覧は複数人可能
- 更新は1日数回程度
- 編集は原則1人ずつ行う
- 編集時はロックファイルを作成する
- ロック中は他ユーザーは閲覧専用で利用する
- DB更新前または起動時にバックアップを行う
- 古いロックは管理者が解除可能とする

## 11. 画面仕様

### 11.1 メイン画面

メイン画面は現行の行動予定表の見た目を可能な限り維持する。

表示項目:

- 氏名
- 行先
- 内線番号
- 備考
- 表示対象は使用中フラグONの社員のみとする

補足表示:

- 氏名欄の色で出社状態を表示する
- 状態札で所在状態を表示する
- 更新日時を確認できるようにする

UI方針:

- 一覧画面は閲覧性を最優先とする
- 余計なボタンは極力表示しない
- 電話応対者が見た瞬間に状況を把握できることを重視する
- 現行のホワイトボードに近い見た目とする

### 11.2 更新画面

更新画面は対象社員の状態を変更するための簡易画面とする。

入力項目:

- 氏名
- 内線番号（表示のみ、編集不可）
- 出社状態
- 所在状態
- 行先
- 備考

更新ルール:

- 氏名は選択式とする
- 氏名選択候補は使用中フラグONの社員のみとする
- 内線番号は社員マスタから取得して表示する
- 所在状態は選択式とする
- 保存時に更新者と更新日時を記録する

### 11.3 管理画面

管理者向けに以下を管理できるようにする。

- 社員マスタ
- 表示順
- 内線番号
- 使用中フラグ
- 状態マスタ（将来対応可）
- 使用中フラグOFFの社員も表示し、再有効化できるようにする

## 12. 電話応対向け表示要件

本システムは外線電話応対時に使われることを前提とする。  
そのため、一覧画面で以下が即座にわかる必要がある。

- 対象者が出社中か退社済みか
- 対象者が在席か不在か
- 不在なら理由が何か
- 必要に応じて行先や補足が何か
- 固定内線番号が何番か

## 14. 入力ルール

- 氏名は必須
- 出社状態は必須

- 退社済み時は氏名欄を赤表示する

## 15. 編集ロック仕様

- 編集開始時に `lock\edit.lock` を作成する
- ロックファイルには以下を記録する
  - 利用者名（Windowsログオンユーザー名）
  - PC名
  - 開始時刻
  - アプリ版数
- ロック中は他ユーザーを閲覧専用モードとする
- 編集中はアプリが5分ごとにロックファイルの更新時刻を更新する
- ロックファイルの更新時刻から60分以上更新されていない場合のみ、期限切れロックとみなす
- アプリ終了時、保存時、キャンセル時にロックを解除する
- 管理者はロック解除機能を利用できる

### 15.1 管理者判定

- 管理者判定はWindowsログオンユーザー名で行う
- 管理者ユーザー名一覧は `config.json` の `admin_users` に保持する
- 一覧に含まれるユーザーのみロック解除機能を利用できる

## 16. バックアップ仕様

- 起動時に共有DB全体で1日1回まで自動バックアップを行う
- 社員マスタ変更前には追加でバックアップを行う
- 通常の状態更新では都度バックアップは行わない
- 当日作成済みバックアップが `backup` フォルダ内に存在する場合は、新たな起動時バックアップを作成しない
- 起動時バックアップ実施判定はバックアップファイル名の日付で行う
- バックアップファイル名は日時付きとする
- 世代数は直近30世代を保持する
- バックアップファイルは `backup` フォルダに保存する

## 17. 更新仕様

- アプリ起動時に `update\latest.json` を確認する
- 新版がある場合は案内メッセージを表示する（初版は案内のみ、自動差し替えは第二版以降）
- 利用者は常に `app\current\Koudouhyo.exe` を起動する
- 新版配布時は管理者が `app\current` を新版へ差し替える
- `app\releases` は版管理・保管用とし、一般利用者は直接起動しない
- 初版の新版案内メッセージは「新版があります。管理者に確認してください。」を基本とする
- DBスキーマ差分がある場合は起動時に移行処理を行う
- 更新失敗時は旧版で起動可能とする

想定する `latest.json` 例:

```json
{
  "version": "1.1.0",
  "path": "\\\\server\\share\\koudouhyo\\app\\releases\\1.1.0\\Koudouhyo.exe",
  "notes": "状態表示改善"
}
```

## 20. 利用者名・更新者の取得方法

- 更新者（updated_by）はWindowsログオンユーザー名を自動取得して保存する
- メイン画面下部の利用者名表示も同じ値を使う
- PC名はロックファイルにのみ保存する

## 21. 退社済み時の画面表示

- 出社状態を退社済みに変更した場合、一覧画面では所在状態表示を「退社済み」として扱う
- 直前の所在状態はDB上保持し、履歴にも記録する

## 22. 所在状態の固定値

初版の所在状態は以下の固定リストとし、管理画面からは変更不可とする。

- 在席
- 外出
- 出張
- 休暇
- 会議
- 直行
- 直帰
- その他

状態マスタ対応は将来拡張機能とする。

## 23. 検索機能の対象

- 検索対象は氏名、行先、備考とする
- 検索結果には使用中フラグONの社員のみ表示する
- 部署は初版の検索対象外とする

## 24. SQLite共有利用の前提

- SQLiteをネットワーク共有上で利用するが、更新頻度が低く、編集ロックを行う前提で運用する
- 同時編集は行わない
- 同時閲覧は許可するが、ロック競合や一時的なアクセス失敗時はリトライまたはメッセージ表示で対応する
- SQLiteのjournal_modeはWALではなくDELETEまたはTRUNCATEを前提とする

## 25. 非機能要件

- 社内Windows PCで動作すること
- 通常操作で大きな待ち時間が発生しないこと
- 一般利用者が迷わず使えること
- 現行運用から見た目を大きく変えないこと
- DB損傷リスクを最小限にすること
