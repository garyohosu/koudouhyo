# QandA.md

MDファイルのレビューで生じた不明点・確認事項と回答。

---

## Q1. ロック期限切れの時間はどれくらいか？

**対象**: SPEC.md §15

**回答**: 初版は **60分**。
更新作業は短時間の想定だが、電話対応や離席で少し放置されることもある。30分だと短すぎて誤解除の可能性がある。

**仕様追記**:
- ロックファイルの更新時刻から60分以上経過した場合、期限切れロックとみなす。
- 管理者は期限切れロックを解除できる。

## Q2. 管理者の識別方法は？

**対象**: SPEC.md §15、UI.md §3.3

**回答**: 初版は **設定ファイルにWindowsログオンユーザー名を列挙**。
パスワード方式は作り込みが増えるため採用しない。

**仕様追記**:
- 管理者判定はWindowsログオンユーザー名で行う。
- 管理者ユーザー名一覧は config.json に保持する。
- 一覧に含まれるユーザーのみロック解除機能を利用できる。

## Q3. updated_by（更新者）はどこから取得するか？

**対象**: SPEC.md §9.2、DB.md §4・§5

**回答**: **Windowsログオンユーザー名を自動取得**で統一。PC名はロック情報にのみ保存する。

**仕様追記**:
- updated_by はWindowsログオンユーザー名を自動取得して保存する。
- メイン画面下部の利用者名表示も同じ値を使う。
- PC名はロック情報にのみ保存する。

## Q4. バージョンアップは自動更新か、案内のみか？

**対象**: SPEC.md §17

**回答**: 初版は **案内のみ**。exe自己更新は第二版以降とする。

**仕様追記**:
- 初版では起動時に latest.json を確認し、新版がある場合は案内メッセージを表示する。
- 自動差し替え更新は第二版以降とする。

## Q7. 退社済みのときの所在状態はどうするか？

**対象**: SPEC.md §10

**回答**: 出社状態を退社済みにした場合、**画面表示上は退社済みを優先**する。直前の所在状態は履歴に保持する。

**仕様追記**:
- 出社状態を退社済みに変更した場合、一覧画面では所在状態表示を「退社済み」とする。
- 直前の所在状態は履歴に保持する。

## Q8. SQLiteをネットワーク共有に置くことのリスク認識はあるか？

**対象**: SPEC.md §5、DB.md §1

**回答**: リスク認識あり。更新頻度が低く、編集ロックを前提とするため、運用でカバーする。

**仕様追記**:
- SQLiteをネットワーク共有上で利用するが、更新頻度が低く、編集ロックを行う前提で運用する。
- 同時編集は行わない。
- 同時閲覧は許可するが、ロック競合や一時的なアクセス失敗時はリトライまたはメッセージ表示で対応する。
- SQLiteのjournal_modeはWALではなくDELETEまたはTRUNCATEを前提とする。

## Q11. バックアップ「更新前に必要に応じて」の条件は？

**対象**: SPEC.md §16

**回答**: 条件を明確化する。通常の状態更新では都度バックアップは行わない。

**仕様追記**:

## Q12. app_configとconfig.jsonの使い分けは？

**対象**: CONFIG.md、DB.md §6

**回答**: 役割を分担する。

- **config.json**: クライアント起動に必要な外部設定（shared_root、管理者ユーザー一覧）
- **app_config**: DB内部で共有されるアプリ設定（schema_version、将来のシステム共通設定）

**仕様追記**:
- config.json はクライアントごとの起動設定を保持する。
- app_config はDB内部で共有すべきアプリ設定を保持する。
- shared_rootのような接続前提情報は config.json に保持し、schema_versionのようなDB依存情報は app_config に保持する。

## Q13. SQLiteの外部キー制約を有効化するか？

**対象**: DB.md §4.3

**回答**: **有効化する**。整合性はDB側でも守る。

**仕様追記**:
- SQLite接続時に毎回 `PRAGMA foreign_keys = ON` を実行する。
- current_status.employee_id および status_history.employee_id は employee_master.id を参照する外部キーとする。
- 社員削除は物理削除せず is_active=0 の論理削除とする。

## 決定事項まとめ

| 項目 | 決定内容 |
|------|---------|
| ロック期限 | 60分 |
| 管理者判定 | config.json のWindowsユーザー名一覧 |
| 更新者取得 | Windowsログオンユーザー名（自動） |
| バージョン更新方式 | 初版は案内のみ |
| 新規社員のcurrent_status | 自動作成（出社中・在席・空欄） |
| 所在状態 | 初版固定値 |
| 退社済み時の表示 | 画面上は退社済み優先表示 |
| SQLite共有利用 | WAL不使用・低頻度更新+ロック前提 |
| 検索対象 | 氏名・行先・備考 |
| app_config記載 | SPEC.mdに追記 |
| バックアップ条件 | 起動時1日1回 + マスタ更新前 |
| config.json/app_config使い分け | 役割分担を明確化 |
| 外部キー制約 | PRAGMA foreign_keys = ON |
| ロック更新方式 | 編集中は5分ごとに更新 |
| current_status の一意性 | employee_id に UNIQUE 制約 |
| is_active=0 の表示 | 管理画面のみ表示 |
| 起動時バックアップ単位 | 共有DB全体で1日1回 |
| アプリ起動場所 | 利用者は `app\\current` 固定 |

## Q14. 編集ロック中に60分を超えた場合、ロックファイル時刻を定期更新するか？

**対象**: SPEC.md §15

**回答**: **更新する**。編集中は 5分ごとにロックファイル時刻を更新する。

**仕様追記**:
- 編集中はアプリが 5分ごとにロックファイルの更新時刻を更新する。
- 60分以上更新されていないロックのみ期限切れとみなす。
- アプリ終了時または保存・キャンセル時にロックを解除する。

---

## Q15. current_status の 1社員1レコードは DB 制約で保証するか？

**対象**: DB.md §4.3、§7

**回答**: **保証する**。`UNIQUE(employee_id)` を付ける。

**仕様追記**:
- `current_status.employee_id` には UNIQUE 制約を付与する。
- これにより、1社員につき現在状態は1レコードのみ保持することをDBでも保証する。
- 更新時は INSERT ではなく、既存レコードに対する UPDATE を基本とする。

---

## Q16. 使用中フラグが OFF の社員はどの画面まで非表示にするか？

**対象**: SPEC.md §9.1、UI.md §2、UI.md §4

**回答**: 提案どおりでよい。

**仕様追記**:
- `is_active=0` の社員は、メイン画面・検索結果・編集候補一覧には表示しない。
- 管理画面では表示し、再有効化できるようにする。

## Q17. 起動時「1日1回」バックアップはクライアント単位か共有DB単位か？

**対象**: SPEC.md §16

**回答**: **共有DB全体で1日1回** とする。

**仕様追記**:
- 起動時バックアップは共有DB全体で1日1回とする。
- 当日作成済みバックアップが `backup` フォルダ内に存在する場合は、新たな起動時バックアップを作成しない。
- 判定はバックアップファイル名の日付で行う。

## Q18. 新版案内後の配布運用は `app\\current` 差し替えか、利用者が release を直接起動するか？

**対象**: SPEC.md §6、§17、FOLDER_STRUCTURE.md

**回答**: 提案どおり、利用者は常に `app\\current` を起動し、管理者が `current` を差し替える運用で統一する。

**仕様追記**:
- 利用者は常に `app\\current\\Koudouhyo.exe` を起動する。
- 新版配布時は管理者が `app\\current` を新版へ差し替える。
- `app\\releases` は版管理・保管用とし、一般利用者は直接起動しない。
- 初版の新版案内は「新版があります。管理者に確認してください。」または「再起動後に最新版を利用してください」と表示する。
